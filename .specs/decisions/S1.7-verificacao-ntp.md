# S1.7 — Verificação de NTP nas duas VPS (runbook)

- **Status:** Verificado — sincronizado nas duas VPS (2026-07-16)
- **Data:** 2026-07-16
- **História:** S1.7 (ops) — issue #11
- **Responsável:** `devops-specialist`
- **Re-checagem:** go-live (S6.3, issue #29)

## Por que isto importa

O `purchased_at` de cada venda **será capturado no plugin** (VPS do jogo) com
`Instant.now()` — o comando do plugin ainda está planejado para o Sprint 2
(S2.4, issue #15) — e a API (VPS `sales.austv.net`) gravará esse timestamp. Se os
relógios das duas máquinas divergirem (clock skew, spec §7), a série temporal e a
apuração ficarão com horários errados — e não há como corrigir depois, porque o
timestamp é a fonte da verdade do evento. Por isso, **antes** de o cutover
confiar no `purchased_at`, é preciso confirmar que **as duas VPS** estão
sincronizadas via NTP, e manter a sincronização persistente (sobrevive a reboot).

> Escopo: **VPS do jogo** (onde roda o plugin/Paper) e **VPS da API**
> (`sales.austv.net`). Rodar o procedimento abaixo em **cada uma**.

## Procedimento

### 1. Verificar estado da sincronização

```bash
timedatectl status
```
Esperado:
- `System clock synchronized: yes`
- `NTP service: active`
- `Time zone: America/Sao_Paulo` (convenção do projeto)

### 2. Medir o offset (skew) — se usar chrony

```bash
chronyc tracking        # olhar os campos: "System time" (desvio do relogio vs.
                        # NTP, em segundos) e "Last offset"/"RMS offset" (skew)
chronyc sources -v      # fontes NTP, stratum e last-sync
```
Se usar `systemd-timesyncd` em vez de chrony:
```bash
timedatectl timesync-status   # "Offset" e servidor NTP em uso
```

### 3. Habilitar/persistir NTP onde faltar

```bash
# systemd-timesyncd (padrão em muitas distros)
sudo timedatectl set-ntp true
sudo systemctl enable --now systemd-timesyncd

# OU chrony
sudo systemctl enable --now chrony    # (chronyd em algumas distros)
```
`enable` garante que sobrevive a reboot. Reconferir com o passo 1.

> **Use apenas um** dos dois — `systemd-timesyncd` **ou** `chrony`, nunca os dois
> ativos ao mesmo tempo (disputam o relógio e geram sincronização conflituosa).
> Se optar pelo chrony, desabilite o timesyncd
> (`sudo systemctl disable --now systemd-timesyncd`), e vice-versa.

### 4. Registrar o resultado

Preencher a tabela abaixo com o medido em cada VPS e commitar a atualização
deste arquivo (ou colar a saída no PR do S1.7).

## Resultados medidos

> Data da medição: 2026-07-16.

| VPS | `synchronized` | Serviço NTP | Offset/skew medido | NTP persistente (reboot) |
|---|---|---|---|---|
| API (`sales.austv.net`, host `austv`) | yes | chrony (stratum 2, `e.st1.ntp.br`) | System time 6.5 µs slow · Last offset −2.66 µs · RMS 22 µs (sub-ms) | serviço ativo |
| Jogo (plugin/Paper, host `austv-5900xt`) | yes | systemd-timesyncd | sincronizado (offset preciso não coletado — `chronyc` ausente; `timedatectl timesync-status` daria o número) | serviço ativo |

**Skew relativo entre as duas VPS:** desprezível para o propósito. Ambas reportam
`System clock synchronized: yes` contra NTP; a VPS da API está sub-milissegundo e
a VPS do jogo sincronizada via timesyncd. A diferença entre relógios NTP-sincados
fica tipicamente na casa de milissegundos — muito abaixo de 1 s, folgado para a
granularidade de evento de venda.

**Observações / ações tomadas:**
- **VPS da API (`austv`):** chrony, `synchronized: yes`, stratum 2, fontes NTP.br
  saudáveis, offset sub-milissegundo — ótimo.
- **VPS do jogo (`austv-5900xt`):** `systemd-timesyncd`, `synchronized: yes`, NTP
  ativo. Fuso em `Etc/UTC` (a convenção do projeto é America/Sao_Paulo) — **sem
  impacto** no `purchased_at`, porque o `Instant.now()` do plugin é um instante
  UTC, independente do fuso local; o que importa é o relógio estar sincronizado.
  Opcional: `timedatectl timesync-status` para o offset numérico exato.
- **Conclusão:** as duas VPS estão sincronizadas via NTP e o serviço está ativo em
  ambas — critério de aceite do #11 atendido. Será re-checado no go-live (S6.3).
